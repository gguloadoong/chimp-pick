## 1. 프로젝트 개요

### 배경
"주식/코인 가격을 예측하는 게임을 만들자"는 아이디어에서 시작. 실제 돈이 아닌 포인트 기반으로, MZ세대가 캐주얼하게 즐길 수 있는 예측 게임을 목표로 함.

### 초기 구조
- 10개 종목 중 유저가 선택 → 바나나코인 베팅 → UP/DOWN 예측 → 결과 확인
- 백엔드 API 기반 설계 (NestJS + PostgreSQL)

### 초기 문제
프로덕션 배포 후 확인: **백엔드 서버가 없어서 게임이 동작하지 않음.** Vercel 서버리스 환경에서 API 서버를 띄울 수 없었음. 이 시점에서 클라이언트 전용 아키텍처로 전환 결정.

## 2. 핵심 의사결정: 게임 메카닉 피봇

### 문제 정의

앱을 직접 플레이해본 후 근본적 문제를 발견:

> "배팅에 초점을 맞추면 안 될 것 같다. 맞출 때마다 승점을 쌓는 형식이고, 그게 리더보드로 나오면 좋겠다."

**기존 메카닉의 문제:**
1. **배팅 = 도박 느낌** — 바나나코인을 걸고 잃는 구조는 지속성이 약함
2. **종목 선택 피로** — 10개 종목에서 고르는 것 자체가 진입 장벽
3. **개인 플레이** — 혼자 하는 게임에서 소셜 요소 부재
4. **리텐션 메커니즘 부재** — "다시 와야 하는 이유"가 없음

### 사고 과정

핵심 질문: "사람들이 이 게임에 자주 들어오는 이유가 뭐가 되어야 하나?"

**가설**:
- 정해진 시간에 라운드가 열리면, "지금 안 하면 놓친다"는 FOMO가 작동
- 다른 사람들의 선택 비율이 보이면, "남들과 다르게" 하고 싶은 심리가 작동
- 소수파가 더 높은 점수를 받으면, 역발상에 대한 보상이 생김

**벤치마크**: Polymarket의 예측 시장 구조를 참고하되, 실제 돈 대신 점수로 진입 장벽을 없앰.

### 해결 방향

| AS-IS | TO-BE |
|-------|-------|
| 유저가 종목 선택 | 시스템이 라운드마다 1개 출제 |
| 바나나코인 베팅 | 승점 누적 |
| 고정 배율 (1.8x) | 소수파 보너스 (적은 쪽 최대 5배) |
| 아무 때나 예측 | 10분 간격 정기 라운드 |

**소수파 보너스 공식**: `점수 = 100 × (다수파비율 / 소수파비율)`
- UP 70% vs DOWN 30% → DOWN 맞추면 233점, UP 맞추면 43점
- 이 공식이 "남들과 다르게 생각하면 보상이 크다"는 핵심 메카닉

### 결과
- Issue #1 → PR #2로 구현, 14파일 변경 (+966 -1259)
- 4개 AI 코드 리뷰어(Gemini, CodeRabbit, Copilot, PR Agent) 통과 후 머지

### 라운드 주기에 대한 고민

> "특정 시간마다 (예를 들어 5분마다? 아니면 10분마다?) 가격을 제공하고, 업/다운을 결정하는 거지."

5분 vs 10분을 고민했을 때:
- **5분**: 코어 유저에겐 좋지만 캐주얼 유저는 못 따라옴
- **10분**: 밸런스 좋음. 1시간에 6라운드
- **데모에서는 30초**: 체험 속도를 위해 짧게. 설정에서 30초/1분/5분 선택 가능하게

결국 "고정 주기"보다 "유저가 선택 가능한 주기"가 답이었음. 하지만 카테고리별로도 달라야 했음 — 재미 질문("치킨 vs 피자?")에 30초는 너무 길고, 시세 예측은 가격 변동을 봐야 하니 30초가 적절.

→ **카테고리별 차등 시간**: 시세 30초, 재미/스포츠 15초, 상식/트렌드 20초

### 소수파 보너스의 유저 심리

> "사람들의 비율에 따라 고득점이 가능해지는 구조로 말이야."

이 한 마디에서 핵심 메카닉이 탄생:
- 다수가 UP을 선택하면 → DOWN을 맞춘 소수가 더 높은 점수
- 게임이론의 "역배" 심리: "남들과 다르게 생각하면 보상이 크다"
- Polymarket과 유사하지만, 실제 돈이 아니니 진입 장벽 없음

하지만 나중에 UX 감사에서 발견: 이 비율이 NPC 시뮬레이션이라 내 선택과 무관했음. → 유저 선택 시 비율 2~5% 변동하도록 수정하여 "내 선택이 게이지에 반영된다"는 피드백 추가.

### 배운 점
> 초기 설계가 틀렸다고 인정하고 빠르게 피봇하는 것이 중요하다. "베팅"이라는 익숙한 프레임에서 벗어나 "예측 경쟁"이라는 본질에 집중하니 게임이 더 재미있어졌다.

## 3. 서비스 본질 정의: 시세 vs 재미

### 문제 정의

개발이 진행되면서 근본적 질문이 떠오름:

> "시세 API가 없거나 느리면 이 서비스의 본질은 어떻게 되는가? 콘텐츠 자체가 폴백이 되어야 하지 않는가?"

### 사고 과정

**핵심 인사이트**: 서비스의 본질은 "정확한 시세"가 아니라 "예측의 재미"

매트릭스로 정리:

| | 시세 정확 | 시세 부정확 |
|---|---|---|
| **예측 재미 있음** | 최고 (목표) | OK (현재) |
| **예측 재미 없음** | 시세 앱이 됨 | 아무것도 아님 |

→ 시세는 "신뢰 부스터"이지 "코어 밸류"가 아님.

### 전략-실행 정합성 문제 발견

팀 미팅에서 전략을 합의하고 실행 플랜을 세웠을 때, **전략과 실행의 괴리**를 직접 발견:

> "의사결정과 실행플랜에 괴리가 있는 것 같은데?"

- 전략: "본질 = 예측 재미, 시세 = 보너스"
- 실행 플랜: Phase B를 "시세 API (핵심)"으로 배치

**수정된 실행 플랜:**
```
Phase A: 콘텐츠 강화 (본질이니까 1순위)
Phase B: 소셜 + 품질 (유저 유입 + 안정성)
Phase C: 시세 API (보너스니까 마지막)
```

### "콘텐츠 폴백"이라는 개념

> "시세 API가 없거나 반응이 느리면 이 서비스의 본질은 어떻게 되는지? 콘텐츠 확장해서 콘텐츠 자체를 폴백해야 되지 않는지?"

이 질문에서 "콘텐츠 폴백"이라는 개념이 등장:
- 보통 폴백은 기술적 개념 (서버 장애 → 캐시 반환)
- 여기서는 **콘텐츠가 폴백**: 시세 API가 죽으면 → 재미/상식 질문 비율이 자동으로 100%로
- 유저는 시세 장애를 인지하지 못함 — 그냥 "오늘은 상식 퀴즈가 많네" 정도

이를 위해 `setLivePriceAvailable(boolean)` 함수로 동적 비율 조정 구현:
- 시세 연결: 시세 40% / 재미 20% / 상식 15% / 스포츠 15% / 트렌드 10%
- 시세 미연결: 재미 30% / 상식 25% / 스포츠 25% / 트렌드 20%

**핵심**: 시세가 죽어도 서비스는 죽지 않는 구조. 이건 기술적 결정이 아니라 서비스 설계 결정.

### 결과
- 콘텐츠 40종 → 120종+ 확장 (재미/상식/스포츠/트렌드 5종 카테고리)
- 동적 질문 비율: 시세 연결 상태에 따라 자동 조정
- Vitest 21건 + GitHub Actions CI 구축

### 배운 점
> 전략적 의사결정과 실행 계획은 반드시 일관성을 검증해야 한다. "보너스"라고 정의해놓고 실행에서 "핵심"으로 배치하면 팀이 혼란스러워진다.

## 4. 유저 여정과 게임 루프 설계

### 첫 진입 유저의 여정

유저가 침팬지픽을 처음 만났을 때 겪는 경험을 설계:

```
랜딩 페이지 → "바로 시작하기" → 온보딩 3단계 → 첫 라운드 → 결과 → "한 판 더?"
```

**랜딩 페이지의 역할**: 3초 안에 "이게 뭔지" 이해시키기.
- 히어로: 침팬지 캐릭터 + "UP or DOWN? 예측하고 점수를 쌓아보세요!"
- 기능 카드 3장: ⏱️ 30초 라운드 / 🧠 3종 카테고리 / 🏆 소수파 보너스
- CTA: "바로 시작하기 🍌" (가입 없이 즉시 플레이)

**온보딩의 딜레마**: 너무 길면 이탈, 너무 짧으면 이해 못함.
- 3단계로 제한: 게임 소개 → 라운드 설명 → 소수파 보너스
- 인터랙티브 튜토리얼은 보류 — 차라리 바로 플레이하게 하고 첫 라운드 자체가 학습

**"한 판 더" 유도**: 한 라운드가 끝난 후 이탈하지 않게 만드는 요소들
- 결과 직후 "다음 라운드!" 버튼 (승리 시) / "복수하기!" (패배 시)
- "내일 다시 오면 출석 보너스!" 예고
- 오늘 미션 진행도 표시 ("2/3 완료! 하나만 더!")

### 게임 루프의 감정 곡선

한 라운드를 "감정 곡선"으로 설계:

```
[평온] → [긴장] → [선택] → [기대] → [절정] → [쾌감/아쉬움] → [동기부여]
  ↑                                                              ↓
  └──────────────────── 다음 라운드 ─────────────────────────────┘
```

| 단계 | 시간 | 유저 감정 | UI/UX |
|------|------|----------|-------|
| 질문 등장 | 0초 | "뭐지?" 호기심 | 질문 카드 + 카테고리 배지 |
| 정보 확인 | 3~5초 | "음..." 분석 | 가격/비율 게이지 |
| UP/DOWN 선택 | 5~10초 | "이거다!" 확신 | 버튼 누름 + 사운드 |
| 결과 대기 | ~종료 | "제발..." 긴장 | 타이머 + 가격 변동 |
| 드럼롤 | 3초 | 두근두근 | 🥁 + 화면 흔들림 |
| 결과 공개 | 1초 | 폭발/탄식 | 🎉 또는 😵 |
| 점수 확인 | 2초 | 만족/아쉬움 | 카운트업 + 전략 힌트 |
| 다음 유도 | - | "한 판 더!" | CTA 버튼 |

**핵심 발견**: "결과 대기" 구간이 게임의 진짜 재미. 시세 예측이면 가격이 실시간으로 오르내리는 걸 보면서 "올라라... 올라라..."하는 긴장감이 핵심. 재미/상식 질문은 이 구간이 없으니 짧게(15~20초) 가야 함.

### 죽은 시간 문제 — 인터랙션 공백

> "사용자가 30초 중 실제로 뭔가 하는 시간은 버튼 한 번 누르는 것뿐"

한 라운드 사이클을 분석해보니:
- 30초 OPEN (하지만 실제 선택은 3~10초)
- 5초 CLOSED (아무것도 안 보임)
- 결과 표시
- 8.5초 BREAK (텍스트만)
- **총 43.5초 중 인터랙션은 한 번**

이건 "게임"이 아니라 "기다림"이었음.

**해결**: CLOSED 3초 + BREAK 3초로 단축 (13.5초 → 6초). 카테고리별 라운드 시간 차등화. 한 사이클이 20~35초로 체감 속도 대폭 향상.

하지만 근본적으로 "선택 후 기다리는 시간"에 뭘 보여줄지가 남은 과제. 현재는 가격 변동 차트가 그 역할을 하지만, 재미/상식 질문에서는 기다릴 이유가 없음.

### 정보 과부하 — 화면 계층화

한 화면에 동시에 표시되던 12개 섹션:
```
헤더 → 챌린지 → 라운드 카드 → 비율 게이지 → UP/DOWN 버튼 → 선택 배너
→ 최근 결과 → 연승 배너 → 데일리 미션 3개 → 시즌 정보 → 출석 보너스 → 푸터
```

모바일에서 핵심인 UP/DOWN 버튼이 화면 중간에 묻혀 스크롤 없이 보이지 않았음.

**정보 계층 재설계:**

| 계층 | 요소 | 항상 노출 |
|------|------|----------|
| **핵심** | 질문 카드 + UP/DOWN 버튼 | ✅ 스크롤 없이 |
| **맥락** | 비율 게이지 + 타이머 | ✅ 질문 카드 내부 |
| **부가** | 미션/시즌/스트릭/최근결과 | ❌ 접이식 서랍 |

"더보기" 토글 버튼 하나로 부가 정보를 숨기고, 접힌 상태에서도 연승 수는 표시 ("미션 · 기록 · 시즌 ▼ · 🔥3연승").

## 5. 메카니즘 검증 — "재미있는가?"

### 문제 정의

기능이 24개까지 늘어났을 때 방향 수정:

> "시세보다는 지금은 이 서비스 메커니즘이 제대로 먹혀드는지 검증하고 고도화하는 게 필요해. 고도화라는 건 콘텐츠의 확장을 말하는 게 아닌, 우리가 생각한 대로 사용자가 재미있게 하냐는 거야."

### "고도화 ≠ 기능 추가"

이 피드백이 중요한 이유: 개발팀은 자연스럽게 "고도화 = 새 기능"으로 해석함. 미션 추가, 칭호 추가, 시즌 추가... 하지만 CEO가 말한 고도화는 완전히 다른 의미:

- ❌ "질문 40종을 120종으로 늘리자" (양적 확장)
- ❌ "챌린지 모드를 추가하자" (기능 추가)
- ✅ "소수파 보너스를 유저가 이해하나?" (메카니즘 검증)
- ✅ "한 판 끝나고 다음 판 하고 싶나?" (리텐션 검증)
- ✅ "긴장 → 결과의 감정 흐름이 설계대로인가?" (경험 검증)

### UX 감사에서 발견한 구조적 결함

| 심각도 | 문제 | 근본 원인 | 유저 경험 영향 |
|--------|------|----------|--------------|
| 🔴 | 퀴즈 결과가 순수 랜덤 | 상식 문제도 동전 던지기 | "아는 게 소용없네" → 이탈 |
| 🔴 | 소수파 비율이 내 선택과 무관 | NPC 비율 고정 | "전략이 의미없네" → 이탈 |
| 🔴 | 죽은 시간 13.5초 | CLOSED + BREAK 과다 | "지루하다" → 이탈 |
| 🟡 | 정보 과부하 | 12개 섹션 한 화면 | "뭘 봐야 하지" → 혼란 |
| 🟡 | 결과 실수 닫힘 | 배경 즉시 닫김 | 점수 못 보고 닫음 → 불만 |
| 🟡 | 재미 질문에 30초 | 3초면 결정 가능 | "왜 이렇게 오래 기다려" → 이탈 |

**근본 원인**: 싱글플레이어인데 멀티플레이어인 척하는 설계. 소수파 비율, 참여 현황이 전부 가짜이니 전략적 깊이가 없었음.

### 해결 — 가짜를 "덜 가짜"로

완전한 멀티플레이어는 서버가 필요하므로 현 단계에서 불가능. 대신:
- **유저 선택이 비율에 반영**: 내가 UP을 누르면 게이지가 2~5% 움직임 → "내 선택이 영향을 주는" 피드백
- **상식 퀴즈에 실제 정답**: 40종에 correctAnswer 추가 → 지식이 의미있음
- **카테고리별 시간 차등**: 시세 30초, 상식 20초, 재미 15초 → 질문에 맞는 리듬

### 배운 점
> 기능 수는 품질 지표가 아니다. "기능 24개 완성"보다 "핵심 1개가 재미있는지"가 중요하다. "재미있는가?"라는 질문에 "기능을 더 추가하면 재미있어진다"는 답은 거의 항상 틀린다.

## 5. 전략과 실행의 정합성

### 문제 정의

전략 리뷰에서 합의한 내용이 실행 플랜에 제대로 반영되지 않는 것을 발견:

> "전략리뷰 내용이 잘 담겨있냐는 말이었어"

### 구체적 괴리

| 전략 합의 | 실행 플랜 | 모순 |
|-----------|----------|------|
| 본질 = 예측 재미 | Phase B를 "핵심"으로 표기 | 보너스라면서 핵심? |
| 콘텐츠가 폴백 | 콘텐츠 강화 Phase가 없음 | 폴백이 되려면 먼저 탄탄해야 |
| 소셜이 유입 핵심 | Phase C (마지막) | 핵심이면 뒤로 빠지면 안 됨 |

### 사고 과정

전략 리뷰에서 나온 핵심 원칙들:
1. Tier 1 (항상 동작) = 콘텐츠 → 이게 먼저
2. 시세는 Tier 2/3 = 보너스
3. "추가보다 기존 강화가 효율적"

이 원칙에 충실한 실행 플랜:
```
Phase A: 콘텐츠 강화 (본질)
Phase B: 소셜 + 품질 (유입)
Phase C: 시세 API (보너스)
```

### 배운 점
> 전략과 실행은 같은 언어로 말해야 한다. 전략에서 "보너스"라고 정의한 것을 실행에서 "핵심"이라고 쓰면 팀 전체가 혼란에 빠진다. PM은 이 정합성을 지속적으로 검증해야 한다.

## 6. 디자인 디테일과 투명성

### 문제 정의

직접 앱을 플레이하면서 발견한 시세 예측 UX 문제:

> "보여준 가격의 기준시점에서 '언제' 기준으로 변화를 비교하는 거야?"

### 발견한 문제들

1. **"현재가" 라벨이 실제로는 "시작가"** — 유저 혼란
2. **24시간 변동률 표시** — 라운드 게임에서 의미 없는 정보
3. **진입가 소수점 노출** (`183,588.482원`) — 비현실적
4. **동가 시 UP 편향** — `>=` 비교로 UP이 유리
5. **30초간 1.3% 변동** — 실제 주식에서 서킷브레이커 수준

### 투명성 의사결정

팀 미팅에서 한물타기(전략가)의 핵심 질문:

> "가짜 시세로 시세 예측 게임이 성립하는가?"

**결론: 유저를 속이지 않는다.** "모의 시세" 배지를 명시적으로 표시.

| 결정 | 근거 |
|------|------|
| "모의 시세" 배지 표시 | 유저 신뢰의 기본 |
| 라운드 기준 변동률 (24h 제거) | 게임에서 의미 있는 정보만 |
| 변동폭 ±0.5% → ±0.1% | 현실적 가격 변동 |
| "시작가"/"현재가" 라벨 분리 | 비교 기준 명확화 |

### 가격 비교 기준시점 — "언제 기준이야?"

직접 플레이하며 스크린샷을 보고 구체적 질문:

> "지금 시점 기준으로 가격이 같을 때는 어떻게 할 거야? 그리고 보여준 가격의 기준시점에서 '언제' 기준으로 변화를 비교하는 거야?"

스크린샷에서 "현재가 183,588.482원"과 "186,014원"이 동시에 보이는데, 어느 것이 비교 기준인지 헷갈림.

**문제 분해:**
1. "현재가"라고 적힌 게 실제로는 "라운드 시작 시점의 가격"(진입가)
2. 큰 폰트의 숫자가 "실시간 변동 가격"
3. 유저 입장에서 "뭐가 기준이고 뭘 보고 판단해야 하지?"

**기대감의 설계**: 유저가 예측한 후 라운드 종료까지 가격이 변동하는 걸 보면서 "올라라... 올라라..." 하는 기대감이 핵심 재미. 이를 위해:
- "시작가"와 "현재가"를 명확히 분리 표시
- 라운드 기준 변동률(+1,014원, +0.55%) 실시간 표시
- 24시간 변동률은 제거 (라운드 게임에서 의미 없는 정보)

### 동가(同價) 문제

가격이 정확히 같으면 UP/DOWN 어느 쪽? 기존 코드는 `exitPrice >= entryPrice`로 동가 = UP. 이건 UP 편향.

**결정**: 동가 시 랜덤 방향. 가격이 안 움직였으면 어느 쪽도 "맞춘" 게 아님. 동전 던지기가 가장 공정.

### 감정 디자인 — 연승과 연패의 재미

> "맞추고 못맞추고 연승하고 연패해도 더 기분좋고 더 기분나쁜 재미를 주면 좋겠어"

이 요청의 핵심: **감정의 진폭을 키우는 것**. 이기면 평범하게 "맞았어요"가 아니라 폭발적으로 기쁘고, 지면 "틀렸어요"가 아니라 우스꽝스럽게 아쉬워야 함.

**승리 감정 에스컬레이션:**
- 일반 적중 (100점 미만): "맞았다! ✨" — 담담한 성취
- 적중 (100점+): "적중! 🎉" — 기분 좋은 확인
- 소수파 대박 (200점+): "대박!! 🎉🎉" — 폭발적 쾌감 + "소수파 역배 성공! 침팬지 천재! 🧠🚀"

**패배 감정 설계:**
- 단순 "0점" 표시가 아니라 상황 맞춤 전략 힌트:
  - UP이 60%+ 였을 때: "소수파를 노려보세요! 적은 쪽이 더 높은 점수!"
  - DOWN이 60%+ 였을 때: "직감을 믿어보세요! 다수파가 항상 옳진 않아요!"
  - 반반이었을 때: "바나나 껍질에 미끄러졌다... 다시 도전!"
- 패배해도 "다음엔 이렇게 해봐"라는 전략적 동기 부여

**참여비율 비주얼 — 원숭이 몰림:**

> "참여비율은 원숭이가 우르르 몰리는 그런 우수꽝스러운 재미"

기존의 밋밋한 막대 게이지 → 원숭이 실루엣 12마리가 UP/DOWN 쪽으로 비율에 따라 몰림. 각각 다른 타이밍으로 바운스 애니메이션. 소수파 쪽에 "보너스!" 강조.

이건 정보 전달이 아니라 **분위기 조성**. "우르르 몰리는" 느낌이 "나도 여기 끼고 싶다" 또는 "저쪽은 함정이야" 같은 심리를 자극.

### 결과 발표의 긴장감 — 드럼롤 시퀀스

결과 발표 순간이 게임의 클라이맥스:
1. **드럼롤** (1.2초) — 🥁 화면 흔들림 + 사운드
2. **결과 공개** (0.8초) — 🎉 또는 😵 큰 이모지로 방향만 먼저
3. **상세 카드** — 점수 카운트업 애니메이션 + 전략 힌트

3.5초 동안은 닫기 버튼 비활성화 — 실수로 터치해서 결과를 못 보는 일 방지.

### 배운 점
> 디자인 디테일은 "예쁜 화면"이 아니라 "정보의 정확성과 감정의 설계"다. 잘못된 라벨 하나가 유저 경험 전체를 무너뜨릴 수 있고, 적절한 감정 피드백이 "한 판 더"의 동기가 된다. 그리고 감정의 진폭이 클수록 게임은 더 재미있어진다 — 이기면 폭발, 지면 아쉬움, 둘 다 "한 판 더"로 이어져야 한다.

## 7. 배포 판단과 유저 검증 전환

### 문제 정의

24개 기능, 17 PR을 머지한 시점에서 전략가의 핵심 질문:

> "기능은 충분하다. 이제 검증 단계다. 이걸 누가 쓰는가?"

### 사고 과정

**기능 추가 관성** vs **유저 검증 필요성**

| 기능 추가 계속 | 배포 + 검증 전환 |
|--------------|----------------|
| 완벽한 제품 출시 | 빠른 학습 |
| 유저 없이 방향 설정 | 유저 피드백 기반 방향 |
| 시간/리소스 낭비 위험 | "충분히 좋은" 시점 판단 |

**결론: 기능 추가를 멈추고 배포 + 유저 검증으로 전환**

DEC-013: "기능 추가 중단, 배포 + 유저 검증 우선"

### 배포 전 마지막 정리
- UI 정리: 핵심 영역(질문+버튼) 집중, 부가 정보 접이식
- 프로덕션 배포: https://chimp-pick.vercel.app

### 다음 단계
1. 5명 유저 테스트 — 5분 플레이 후 피드백
2. 핵심 메트릭 정의 — 세션 시간, 라운드 참여 수, 재방문율
3. 피드백 기반 개선 루프

### 배운 점
> "충분히 좋은" 시점을 판단하는 것이 PM의 핵심 역량이다. 완벽함을 추구하며 배포를 미루는 것보다, 유저 앞에 빨리 내놓고 학습하는 것이 더 가치 있다.

## 8. 확장 비전과 카테고리 설계

### 문제 정의

CEO의 핵심 비전:

> "이런 걸 비단 주식과 코인 가격말고도 다양한 분야의 예측게임으로 확장하고자 해."

### 사고 과정

단순히 질문을 늘리는 것이 아니라, **확장 가능한 구조**가 필요:

1. `QuestionProvider` 인터페이스 설계 — 카테고리별 질문 생성/판정을 독립 모듈로
2. 카테고리별 특성 차이 — 시세 질문은 가격 비교, 상식은 정답 판정, 재미는 의견 기반
3. 비율 — 시세 연결 시 40%/미연결 시 30% + 나머지 카테고리 균등 배분

**확장 결과**: 3종(시세/재미/상식) → 5종(+스포츠/트렌드)으로 120종+ 질문

### 배운 점
> 확장성은 "나중에 추가하면 되는 것"이 아니라 "처음 설계할 때 인터페이스를 분리해두는 것"이다. `QuestionProvider` 추상화가 있었기에 새 카테고리를 추가할 때 게임 엔진 코드를 수정할 필요가 없었다.

## 9. 시세 API — 기술과 우선순위의 분리

### 문제 정의

> "실제 시세 API를 사용한다면 어떤 걸 써야 문제없을지, 주식은 너무 제한적일 테고, 코인 쪽도 실시간성이 쉽지는 않아 보이는데 근본적 고민 병행해줘."

### 사고 과정

| API | 실시간성 | 비용 | CORS | 결론 |
|-----|---------|------|------|------|
| Upbit WebSocket | ms 단위 | 무료 | 브라우저 직접 가능 | ✅ 1순위 |
| Binance REST | 5초 폴링 | 무료 | 허용 | ✅ 폴백 |
| CoinGecko | 30초+ | 무료 | 허용 | ❌ 너무 느림 |
| KIS (주식) | 실시간 가능 | 무료(계좌 필요) | 차단 | ❌ 서버 필요 |
| 네이버/Yahoo | 15분 지연 | 무료 | 차단 | ❌ 의미 없음 |

**서버리스(Vercel) 제약**:
- WebSocket 서버 호스팅 불가
- 하지만 Upbit WS는 **클라이언트에서 직접 연결 가능** → 서버 비용 0

**결론**: 코인은 Upbit WS로 실시간 가능, 주식은 당분간 보류 (ROI 낮음).
하지만 CEO 판단으로 시세보다 메카니즘 검증이 우선 → 시세 API는 Phase C로 밀림.

### 배운 점
> 기술적으로 가능한 것과 지금 해야 하는 것은 다르다. Upbit WS 연동은 기술적으로 가능하지만, 핵심 메카니즘이 검증되지 않은 상태에서 "진짜 시세"를 붙이는 건 우선순위 오류다.

## 문서 운영 원칙

이 문서는 **살아있는 문서**입니다.

### 기록 시점
- CEO가 문제를 제기하거나 방향을 수정할 때
- 팀 미팅에서 의사결정이 내려졌을 때
- 직접 플레이하며 UX 문제를 발견했을 때
- 전략과 실행 사이의 괴리를 발견했을 때
- 배포/중단/피봇 등 큰 판단이 있었을 때

### 기록 형식
매 항목은 아래 흐름을 따릅니다:
1. **문제 정의** — 무엇이 문제인지, 왜 문제인지
2. **사고 과정** — 어떤 가설을 세웠고, 어떤 선택지가 있었는지
3. **해결 방향** — 왜 이 방법을 선택했는지 (trade-off 포함)
4. **결과** — 구체적으로 무엇이 바뀌었는지
5. **배운 점** — PM/PO로서 어떤 인사이트를 얻었는지

### 다른 프로젝트에 적용하려면
아래 지시문을 사용하세요:

> "이 프로젝트에서 내가 내리는 의사결정, 문제 제기, 방향 수정을 포트폴리오 케이스 스터디로 기록해줘. `.project/portfolio/README.md`에 '문제 정의 → 사고 과정 → 해결 방향 → 결과 → 배운 점' 형식으로 누적해줘. 빅테크 PM/PO 채용에서 요구하는 역량(문제 발견, 가설 수립, 우선순위 결정, 전략-실행 정합성, 유저 중심 사고)이 드러나도록 작성해줘. 단순 작업 로그가 아니라 내 사고의 흐름이 보여야 해."
